  • Introduction

    This plan specifies the communication of the test approach to "Tic Tac Toe Game". It includes all objectives, scope, test schedule, and approach. This document will clearly identify what test deliverables will be.

    In modern society, we have a various of users and they have a various of devices. This test plan will also provide information about how well is the game supported over all the platform.

    The whole testing will mainly focus on three components, Unit testing, Integration, and System Testing. The details of our test will be further defined in approach section and in level specific plans.

  • Test Items

    The test in this document will include three major components, including game board layout which will also includes basic game logic, check win functionality, button functionality which includes starting new game function, save a game function, load a game function and exit the game.

  • Features To Be Tested

    A feature that will be tested include game board layout, game logic, check the win conditions, load a game, save a game, start a new game and exit a game.

    Also, pop up proper information when a user clicks on certain buttons will also be tested.

  • Features Not To Be Tested

    Tic Tac Toe is a really small, yet complex system. However it does not consume much computing power, in this case, the judgement on its algorithm is not that important. There may be good algorithms than run faster than current one, but the result for end users is unrecongnizable. In this case, the algorithm will not be tested.

  • Approach

    For testing the whole system, we will be dividing testing into four major components, Unit testing, Intergration testing, System testing and acceptance testing.

    Arguments for tests will only be the mouse clicks. The whole system does not require any keyboard input or other input devices. So base on this argument, if we receive expected results, we will consider it is passed.

    It is expected that the appropriate symbol (cross or circle) will appear in the selected slot on the game board. When a player could get 3 of the symbols on a straight line (horizontal, vertical or diagonal), there will be proper information displayed to the users. Save button is expected to save current game board and load button is expected to recover the saved game into the current game board. In addition, any operation that will interrupt current game will arouse warning texts.

    To perform this test, there will be no additional dependencies. The test could be tested on any devices that support browsers and javascript.

  • Item Pass/Fail Criteria

    Here will specify the conditions that must be satisfied. Critical functions that must work in the system should be identified. If there are any of the components fails, the system testing must be stopped.

    All failures will be evaluated and categorized into several categories. By defining failures into categories and then identify if the test has been passed or not can give testers and decision makers a better overlook of the whole process of the project.

    • Category A: Serious system failure. Any Category A failure will cause the whole system test to stop.
    • Category B: Functionality failure. Category B failures are the failures that show the code is not working as we expected. For example, click on the board, but the symbol of the player does not change.
    • Category C: UI failure. This category holds all the interface failure. For instance, if the elements in the game do not show or show in the wrong place, it will be categorized here. C Failure generally will not cause the system test to stop.
    • Category D: failures might be areas where minor problems exist with the system.

    For unit test level:

    • all test cases should be completed.
    • 90% of the test cases should be able to return expected results or be able to provide with the defects information.
    • Test coverage must be at least 90%
    • No Category A failure
    • Category B has less than 3 failure cases

    For Integration test level:

    • all test cases should be completed.
    • 90% of the test cases should pass
    • All test cases associated with critical functionality must pass.
    • Test coverage must be at least 90%
    • All medium and high severity defects must be fixed

    For System test level:

    • No Category A failure
    • Category B has no or 1 failures
    • Less than 3 failure in Category C
    • No more than 2 failure in Category D

  • Suspension Criteria And Resumption Requirements

    Some criteria outlined in the previous section could be used to decide when to suspend the testing.

    Suspension Criteria Resumption Requirements
    If any of the functionality fail and being categorized in Category A, the whole test will be suspended Testing will be resumed if the programmers can downgrade the failure from Category A to C or D or completely fix it
    Assigned test resources are not available when needed by the team, the test procedure will be halted Testing will be resumed one the resources are available
